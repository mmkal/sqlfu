# Lint stale generated queries

## Status Summary

Spec is ready after autonomous grilling. No implementation has started yet. Main decisions: generated query wrappers get a per-file SHA-256 source hash comment, and the lint plugin reports missing or stale wrappers from `.sql` file linting.

## Checklist

- [ ] Add a generated wrapper metadata comment containing a full SHA-256 hex hash of the exact source `.sql` file bytes.
- [ ] Add `sqlfu/generated-query-freshness` to the lint plugin.
- [ ] Make the rule run for standalone `.sql` files through the existing SQL processor virtual file path.
- [ ] Report a clear lint error when the expected generated wrapper is missing, including when `.generated/` does not exist.
- [ ] Report a clear lint error when the generated wrapper hash is missing or differs from the current source hash.
- [ ] Add the rule to `sqlfu.configs.recommended` in the `**/*.sql/**/*.js` block.
- [ ] Document the new rule in the lint plugin docs.
- [ ] Cover the behavior with TDD-style integration tests, one behavior at a time.

## Desired Behavior

When a user edits `sql/<name>.sql`, linting that file should fail until `sqlfu generate` has refreshed `sql/.generated/<name>.sql.ts`.

Generated query wrappers should include a stable metadata comment near the top, for example:

```ts
// Generated by `sqlfu generate`. Do not edit.
// sqlfu-source-hash: <sha256-of-source-sql-bytes>
```

The lint rule should:

- Find the sqlfu project root using the existing `sqlfu.config.*` discovery.
- Resolve the queries directory using the existing cheap config parser and `sql` fallback.
- For each linted source `.sql` file outside `.generated`, derive the wrapper path under `.generated/<relative>.sql.ts`.
- Compute SHA-256 over the current source file bytes.
- Report when the wrapper file does not exist.
- Report when the wrapper exists but does not contain the expected `sqlfu-source-hash` comment.
- Report when the wrapper hash differs from the current source hash.
- Avoid autofix. Running generation can require schema analysis and writes multiple files.

## Out of Scope

- Do not make ESLint run `sqlfu generate`.
- Do not check `tables.ts`, `index.ts`, migration bundles, or schema freshness.
- Do not execute arbitrary `sqlfu.config.ts` from the lint plugin.
- Do not introduce a central generated manifest unless per-wrapper metadata proves unworkable.
- Do not redesign the existing SQL processor.

## Guesses and Assumptions

- Exact source bytes are the right hash input because the rule answers whether generation happened after the file changed, not whether the SQL semantics changed.
- A full SHA-256 hash is preferable to a short hash because it avoids collision explanations in a generated metadata contract.
- Strict reporting when `.generated/` is absent is acceptable because users can disable this rule if they only want SQL formatting.
